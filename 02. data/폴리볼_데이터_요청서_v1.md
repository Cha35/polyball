## 요청 1. 응모권 발급 및 소진 일별 집계

### 목적
"예측은 하는데 응모를 안 하는 유저"가 응모권이 없어서인지, 있는데 안 쓰는 건지 구분

### 지표 정의

| 지표명 | 정의 | 집계 기준 |
|---|---|---|
| `daily_issued_total` | 해당 일자에 발급된 응모권 총 수 | 발급 이벤트 발생일 기준 |
| `daily_issued_pred` | 예측 적중 보상으로 발급된 응모권 수 | 발급 사유 = 예측 보상 |
| `daily_issued_ad` | 광고 시청 보상으로 발급된 응모권 수 | 발급 사유 = 광고 시청 (앱 전용) |
| `daily_issued_other` | 기타 사유 발급 수 (이벤트 등) | 위 둘 외 모든 발급 |
| `daily_used_total` | 해당 일자에 소진된 응모권 총 수 | 응모 제출일 기준 |
| `daily_issued_users` | 응모권을 1장 이상 발급받은 유저 수 | 유니크 user_id |
| `daily_used_users` | 응모권을 1장 이상 소진한 유저 수 | 유니크 user_id |
| `daily_balance_users` | 잔고 > 0인 유저 수 (당일 기준 스냅샷) | 누적 발급 - 누적 소진 > 0 |
| `daily_balance_avg` | 잔고 보유 유저의 평균 잔고 | SUM(잔고) / 잔고>0 유저 수 |

### 요청 형식

```
date, daily_issued_total, daily_issued_pred, daily_issued_ad, daily_issued_other,
daily_used_total, daily_issued_users, daily_used_users, daily_balance_users, daily_balance_avg
```

### 기간
2026-03-26 ~ 2026-06-03 (일별, 70행)

### 분석에서 볼 것
- `daily_issued_users` vs `daily_used_users` 격차 → 발급 받고도 안 쓰는 유저 규모
- `daily_balance_users` 추이 → 응모권 누적 보유자가 늘고 있는지 줄고 있는지
- `daily_issued_ad` 추이 → 광고 시청 활성도 (앱 사용 강도)
- 월요일 직전 `daily_balance_avg` vs 월요일 `daily_used_total` → 쌓아두기 패턴 검증

## 요청 2. 당첨/미당첨 결과 후 유저 행동

### 목적
미당첨 경험이 이탈의 직접 원인인지 확인 (특히 어린이날 응모 결과 이후)

### 지표 정의

**기준: 응모 결과가 발표된 유저를 당첨/미당첨으로 나눠 이후 행동 추적**

| 지표명 | 정의 | 집계 기준 |
|---|---|---|
| `result_date` | 응모 결과 발표일 | 당첨 여부 확정 처리 일자 |
| `result_type` | 당첨 / 미당첨 | |
| `user_count` | 해당 결과를 받은 유저 수 | 유니크 user_id |
| `revisit_d7` | 결과 발표 후 7일 내 재방문 유저 수 | result_date 기준 +7일 이내 로그인 or 활동 |
| `revisit_d30` | 결과 발표 후 30일 내 재방문 유저 수 | result_date 기준 +30일 이내 |
| `reentry_d7` | 결과 발표 후 7일 내 재응모 유저 수 | 응모 이벤트 발생 유저 |
| `reentry_d30` | 결과 발표 후 30일 내 재응모 유저 수 | |

### 요청 형식 — 결과 발표 회차별

```
result_date, result_type, user_count, revisit_d7, revisit_rate_d7(%),
reentry_d7, reentry_rate_d7(%), revisit_d30, reentry_d30
```

### 어린이날 특정 코호트 별도 요청

```
대상: 2026-04-20 ~ 2026-04-26 사이에 5/5~5/10 경기에 응모한 유저
분리: 당첨자 / 미당첨자
추적: 결과 발표일(추정 4/27) 이후 7일, 14일, 30일 재방문/재응모율
```

### 분석에서 볼 것
- 미당첨 재방문율 vs 당첨 재방문율 격차
- 어린이날 미당첨 코호트의 이탈율이 다른 회차 대비 높은지
- → 미당첨 리텐션 장치 필요성 판단

## 요청 3. 앱 설치 여부 × 핵심 행동 비교

### 목적
결과 확인(앱 전용)을 못 하는 웹 전용 유저가 응모에서 이탈하는지 확인

### 유저 분류 정의

| 분류 | 정의 |
|---|---|
| `web_only` | 가입 후 앱 미설치 상태로 조회 시점까지 유지 |
| `app_installed` | 웹 가입 후 앱 설치 완료한 유저 |
| `app_first` | 앱으로 먼저 유입된 유저 (있다면) |

### 지표 정의

| 지표명 | 정의 |
|---|---|
| `user_count` | 분류별 유저 수 |
| `entry_rate` | 가입 후 1회 이상 응모한 비율 = 응모경험자 / 전체 |
| `entry_avg_per_user` | 1인당 평균 응모 횟수 (기간 내 누적) |
| `reentry_rate` | 첫 응모 이후 2회 이상 응모한 비율 |
| `app_install_day` | 웹 가입일로부터 앱 설치까지 경과 일수 (중앙값, 평균) |
| `entry_before_install` | 앱 설치 전 응모 경험 유저 수 |
| `entry_after_install` | 앱 설치 후 응모 참여 증가 여부 (설치 전 vs 후 응모율) |

### 요청 형식

```
user_type, user_count, entry_rate(%), entry_avg_per_user,
reentry_rate(%), avg_days_to_install
```

### 분석에서 볼 것
- `web_only` 유저의 `entry_rate`가 `app_installed` 대비 유의미하게 낮은지
- 앱 설치 전후 응모율 변화 → 앱 설치가 응모 행동을 유발하는지

## 요청 4. 응모권 발급 사유별 유저 행동 차이

### 목적
광고 시청으로 추가 응모권을 버는 유저와 아닌 유저의 응모 강도 차이 확인

### 유저 분류 정의

| 분류 | 정의 |
|---|---|
| `ad_viewer` | 조회 기간 내 광고 시청으로 응모권을 1회 이상 발급받은 유저 |
| `non_ad_viewer` | 광고 시청 응모권 발급 이력 없는 유저 |

### 지표 정의

| 지표명 | 정의 |
|---|---|
| `user_count` | 분류별 유저 수 |
| `avg_entry_count` | 1인당 평균 응모 횟수 |
| `avg_issued_ad` | 광고 시청 유저의 평균 광고 시청 횟수 |
| `entry_rate` | 응모 참여율 |
| `ad_to_entry_rate` | 광고 시청 후 동일 경기 응모 전환율 (광고 시청 이벤트 → 응모 이벤트, 동일 세션 or 당일) |

### 요청 형식

```
user_type, user_count, avg_entry_count, entry_rate(%),
avg_issued_total, avg_issued_ad, avg_issued_pred
```

### 분석에서 볼 것
- `ad_viewer`의 응모 횟수가 유의미하게 높은지
- 광고 시청이 응모 행동 강화에 실제로 기여하는지

## 요청 5. 가입 코호트별 리텐션

### 목적
어느 시기 가입자가 얼마나 오래 남는지, 마케팅 채널별 유입 품질 차이

### 정의

| 지표명 | 정의 |
|---|---|
| `cohort_week` | 가입 주차 (예: 2026-W13) |
| `cohort_size` | 해당 주에 가입한 유저 수 |
| `retention_d1` | 가입 다음날 재방문율 = D1 방문자 / cohort_size |
| `retention_d7` | 가입 후 7일 내 재방문율 |
| `retention_d14` | 가입 후 14일 내 재방문율 |
| `retention_d30` | 가입 후 30일 내 재방문율 |
| `entry_d7` | 가입 후 7일 내 응모 참여율 |
| `entry_d30` | 가입 후 30일 내 응모 참여율 |

**재방문 정의:** 가입일과 다른 날짜에 로그인 또는 예측/응모/퀴즈 이벤트 발생

### 요청 형식 — 코호트 테이블

```
cohort_week, cohort_size, retention_d1(%), retention_d7(%),
retention_d14(%), retention_d30(%), entry_d7(%), entry_d30(%)
```

### 추가 요청 — 채널별 분리 (가능하면)

```
cohort_week, channel, cohort_size, retention_d7(%), entry_d30(%)
```
채널: paid_myseatcheck / instagram_fanpage / round / organic 등 주요 유입 채널

## 요청 시 참고 사항

1. **집계 기준 시간대**: KST 기준 00:00~23:59 (에어브릿지와 동일하게)
2. **중복 제거**: 유저 수는 모두 유니크 user_id 기준
3. **탈퇴 유저 포함 여부**: 분석 목적상 탈퇴 유저도 포함 (이탈 원인 분석이므로)
4. **최소 샘플 기준**: 유저 수 10명 미만 셀은 '-' 처리 요청
5. **파일 형식**: CSV 또는 xlsx
