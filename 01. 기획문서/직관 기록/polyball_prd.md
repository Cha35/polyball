# PART 1. MVP

---

## 1. 배경 및 목적

### 1-1. 배경

현재 폴리볼은 승부 예측 및 응모 기반의 직관 티켓 당첨 서비스를 운영하고 있다. 유저는 폴리볼을 통해 예측에 참여하고, 당첨 시 실제 경기를 관람하는 경험을 얻는다. 그러나 이 경험은 당첨-확인-직관으로 이어지는 단발성 여정에서 끝나며, 이후 유저가 폴리볼로 돌아올 지속적인 이유가 부족하다.

동시에 KBO 직관 기록 시장에서는 야구보구, 직관일기, 나의 야구 일지 등 다수의 전문 앱이 운영되고 있으나, 1인 개발 기반의 서버 불안정 문제와 기능의 파편화가 유저의 페인포인트로 확인된다. 유저들은 더 안정적이고 폴리볼의 승부 예측과 연결된 통합된 기록 경험을 원하고 있다.

### 1-2. 목적

본 기능은 세 가지 목적을 달성하기 위해 기획된다.

**첫째, 유저 리텐션 강화**
직관/집관 기록이 쌓일수록 유저가 폴리볼을 떠나기 어려운 구조를 만든다. 기록은 폴리볼에만 존재하는 유저의 자산이 된다.

**둘째, 승부 예측과의 시너지 창출**
예측 참여 → 직관/집관 → 기록 → 통계 확인으로 이어지는 팬 라이프사이클을 폴리볼 안에서 완결시킨다. 예측과 기록이 연결된 경험은 타 서비스가 제공할 수 없는 폴리볼만의 차별점이다.

**셋째, 내 지갑의 활성화**
현재 당첨자에게만 의미 있는 '내 지갑' 영역을 모든 유저가 지속적으로 방문하는 공간으로 전환한다. 직관 기록 카드, 배지, 당첨 티켓이 쌓이는 컬렉션 공간으로 재정의한다.

### 1-3. 목표 지표 (예시 — 수치는 추후 정의)

| 지표 | 방향 |
|---|---|
| 직관 기록 기능 활성 유저 비율 | 증가 |
| 기록 기능 진입 후 재방문율 | 증가 |
| 내 지갑 탭 방문 빈도 | 증가 |
| 승부 예측 참여 → 기록 연동 전환율 | 신규 측정 |

## 2. 타겟 유저 정의

MVP에서 주요하게 설계해야 할 유저 유형은 세 가지다.

### 유저 A — 열성 기록러
- **특성:** KBO 직관 3년 이상, 기존 기록 앱 다수 사용 경험 보유
- **니즈:** 데이터의 영구적 보존, 디테일한 기록, 예쁜 아카이빙 결과물
- **페인포인트:** 기존 앱의 서버 불안정으로 인한 데이터 유실 경험
- **폴리볼에서 원하는 것:** 안정적인 서버 + 승부 예측과 연결된 통합 기록

### 유저 B — 야구 팬 / 기록 무관심
- **특성:** 야구 팬이지만 기록 앱 사용 경험 없음
- **니즈:** 최소한의 노력으로 최대한의 결과
- **페인포인트:** 수동 입력 피로, 기록의 동기 부재
- **폴리볼에서 원하는 것:** 아무것도 하지 않아도 기록이 자동으로 쌓이는 경험

### 유저 C — 기록 친숙 / 야구 입문
- **특성:** 다꾸, 영화 기록 등 아카이빙 취미 보유. 야구 입문 1년 이내
- **니즈:** 야구 용어 몰라도 채울 수 있는 기록, 감성적인 결과물
- **페인포인트:** 기존 앱의 야구 전문 용어 기반 UI로 인한 진입 장벽
- **폴리볼에서 원하는 것:** 경험과 감정 중심의 기록, 구장 방문 수집 경험

> **라이트 유저 (서브 타겟)**
> 야구 관심은 있으나 깊이 빠져있지 않은 유저. 자동화, 소셜 연결, 티켓북 개념의 수집 감각으로 자연스럽게 유입 가능. MVP에서 직접 설계하기보다, 이들이 거부감 없이 쓸 수 있도록 언어와 온보딩을 배려하는 방식으로 접근한다.

## 3. 유저 시나리오

### 시나리오 1 — 직관 기록 (핵심 플로우)

```
경기 당일 구장 도착
→ 폴리볼 앱 실행
→ GPS 인증 (구장 반경 내 자동 감지)
→ "오늘 [구장명]에서 직관 기록을 시작할까요?" 팝업
→ 확인 터치 → 기록 시작 (날짜/구장/팀 자동 완성)
→ 경기 진행 중 (앱 종료 가능)
→ 경기 종료 후 "기록이 완성됐어요" 푸시 알림
→ 스코어 자동 반영 → 사진/소감 선택 입력
→ 기록 카드 생성 → 내 지갑에 저장
```

### 시나리오 2 — 집관 기록 (예측 연동 플로우)

```
경기 시작 전 승부 예측 참여 (기존 기능)
→ 경기 종료
→ "오늘 [팀] 경기 집관 기록 남길게요?" 푸시 알림
→ 확인 터치 → 스코어 + 예측 결과 자동 반영된 기록 카드 생성
→ 내 지갑에 저장
```

### 시나리오 3 — 응모 당첨 직관 기록 (특별 플로우)

```
응모 당첨 후 직관 당일 구장 도착
→ GPS 인증
→ "폴리볼 당첨으로 오신 특별한 날이에요!" 화면 노출
→ 응모 이벤트명 + 도전 횟수 자동 표기
→ 기록 완성 → 일반 카드와 시각적으로 구분된 특별 카드 생성
→ 내 지갑에 저장
```

### 시나리오 4 — 내 지갑 탐색

```
마이 메뉴 → 내 지갑 진입
→ 요약 정보 확인 (기록 카드 N장 · 배지 N개 · 티켓 N장)
→ 타임라인 뷰에서 최근 기록 확인
→ 특정 카드 탭 → 카드 상세 확인
→ 카드 공유 (고유 URL 기반)
```

## 4. 기능 범위

### In Scope — MVP에서 만드는 것

| 기능 | 상세 |
|---|---|
| GPS 기반 직관 인증 | 구장 반경 내 위치 감지, 자동 기록 시작 |
| 직관 기록 카드 생성 | 날짜/구장/팀/스코어 자동 완성 + 사진·소감·동행인 선택 입력 |
| 집관 기록 (예측 연동) | 예측 참여 이력 기반 경기 종료 후 원탭 기록 |
| 응모 당첨 카드 특별 처리 | 이벤트명·도전 횟수 자동 표기, 시각적 구분 |
| 직관 승률 통계 | 기록 5개 이상 시 오픈, 오픈 전 프로그레스 넛지 |
| 내 지갑 1단계 리뉴얼 | 타임라인 뷰, 요약 정보, 카드 고유 번호 부여, 공유 URL |
| 온보딩 | "오늘부터 기록 시작" 프레이밍, 라이트 유저 진입 장벽 최소화 |

### Out of Scope — MVP에서 만들지 않는 것

| 기능 | 이유 |
|---|---|
| 감성 모드 / 기록 모드 이중 구조 | UX 설계량 과다, 유저 반응 확인 후 결정 |
| 구장 스탬프 지도 | 지도 기능 개발 복잡도, v1.5로 이동 |
| 과거 기록 소급 입력 | 플로우 복잡도, v1.5로 이동 (출시 후 1개월 내) |
| 카드 등급 세분화 | 이벤트 운영 데이터 축적 후 설계, v2 |
| 집관 실시간 퀴즈 인증 | 보상 연동 시점에 강화, v2 |
| 카드 홀로그램 인터랙션 | 개발 리소스, v2 |
| 시즌 Wrapped | 시즌 마무리 시점 출시, v2 |
| 그룹 소셜 / 예측 대결 | 직관 기록 범위 초과, 별도 기획 |
| NFT 실제 발행 | 장기 로드맵, v3 이후 |

## 5. 기능별 요건 정의

### 5-1. GPS 기반 직관 인증

**목적:** 유저가 실제로 구장에 있음을 확인하고, 기록을 자동 시작하는 트리거

**요건**
- KBO 10개 구장 좌표 사전 등록
- 유저 위치가 구장 반경 500m 이내일 때 인증 성공으로 처리
- 인증 성공 시 해당 날짜의 경기 데이터와 자동 매핑
- 당일 해당 구장 경기가 없는 경우 예외 처리 필요
- GPS 신호 불안정 시 재시도 안내 제공

**예외 케이스 (별도 정의 필요)**
- 우천 취소: "경기는 취소됐지만, 구장에 간 기록은 남아요" 처리
- 더블헤더: 두 경기 중 선택하여 기록 가능
- 경기 연장: 스코어 확정 후 자동 반영

**플랜 B (KBO 데이터 연동 불가 시)**
- 경기 일정 파싱으로 날짜/팀 자동 완성
- 스코어는 유저 직접 입력으로 임시 처리
- 데이터 연동 확보 후 자동화로 전환

**티켓 스캔 (부가 옵션)**
- 모바일/종이 티켓 스캔 시 좌석 번호 자동 입력
- GPS 인증의 보완 수단으로 제공, 필수 아님

---

### 5-2. 직관 기록 카드 생성

**목적:** 직관 경험을 유저의 카드 자산으로 전환

**자동 완성 항목 (유저 입력 불필요)**
- 경기 날짜
- 구장명
- 홈팀 / 원정팀
- 최종 스코어 (경기 종료 후 자동 반영)
- 승패 결과

**선택 입력 항목 (유저가 추가 가능)**
- 사진 (최대 N장, 장수 제한 정책 별도 정의)
- 한 줄 소감 (텍스트)
- 동행인 태그
- 날씨 (이모지 셀렉터)
- 기분 (이모지 셀렉터)

**카드 설계 원칙**
- 선택 입력이 하나도 없어도 카드가 완성된 것처럼 보여야 함
- 빈 항목이 시각적으로 "비어있다"는 느낌을 주지 않아야 함
- 직관 카드 / 집관 카드 시각적 구분 (구장 배경 vs 홈 배경 텍스처)
- 카드에 고유 발행 번호 부여 (예: #00847)

**카드 공유**
- 카드별 고유 URL 생성 (예: polyball.com/card/00847)
- 퍼블릭 페이지에서 카드 정보 및 소유자 확인 가능
- 이미지로 저장 / 링크 공유 모두 지원

---

### 5-3. 집관 기록 (예측 연동)

**목적:** 승부 예측 참여를 집관 인증의 간접 근거로 활용, 최소 마찰로 기록 생성

**트리거 조건**
- 해당 경기에 승부 예측 참여 이력이 있는 유저
- 경기 종료 후 자동 푸시 알림 발송

**기본 자동 완성 항목**
- 경기 날짜, 팀, 최종 스코어
- 예측 참여 결과 (성공 / 실패)

**집관 카드 특성**
- 직관 카드와 동일한 선택 입력 항목 제공
- 카드에 "집관" 표기 및 예측 결과 자동 연동
- 예측 성공 + 집관 승리 시 별도 표기

**집관 기록 없이 예측만 한 경우**
- 알림 무시 시 기록 미생성 (강제 아님)
- 이후 마이 기록에서 소급 생성 불가 (MVP 기준)

---

### 5-4. 응모 당첨 카드 특별 처리

**목적:** 폴리볼이 만들어준 경험임을 기록으로 각인, 당첨 경험의 감동 극대화

**자동 연결 조건**
- 응모 이벤트 생성 시 경기 정보 태깅 필수 (운영 프로세스 변경 필요)
- 당첨 유저가 해당 경기 날 GPS 인증 시 자동 감지

**카드에 추가되는 정보**
- 이벤트명
- 응모 도전 횟수
- 당첨 발표 타임스탬프

**시각적 구분**
- 일반 직관 카드와 다른 카드 프레임/배경 디자인
- "폴리볼이 만들어준 날" 명시
- MVP에서는 시각 구분 + 정보 표기까지. 등급 세분화는 v2

**미당첨 유저 처리**
- 응모 이력 자체를 타임라인에 표기
- 응모 횟수 누적 배지 제공 (예: "응모 10회 도전자")

---

### 5-5. 직관 승률 통계

**목적:** 개인화된 데이터로 유저의 흥미와 리텐션 강화

**노출 조건**
- 기록 5개 이상 시 통계 탭 오픈
- 5개 미만 시 "기록 N개 더 쌓으면 내 승률이 열려요" 프로그레스 넛지 노출

**제공 통계 항목 (MVP)**
- 직관 승률
- 집관 승률
- 직관 vs 집관 승률 비교
- 가장 많이 간 구장
- 총 직관 / 집관 횟수

---

### 5-6. 내 지갑 1단계 리뉴얼

**목적:** 당첨자 전용 공간 → 모든 유저의 팬 경험 컬렉션 공간으로 전환

**지갑에 담기는 아이템 종류**

| 종류 | 항목 |
|---|---|
| 티켓류 | 응모 당첨 티켓 (기존 유지) |
| 기록 카드류 | 직관 기록 카드, 집관 기록 카드, 응모 당첨 직관 카드 |
| 배지류 | 응모 도전 배지 (MVP), 그 외 배지는 v2 |

**지갑 화면 구조**
- 상단: 요약 정보 (기록 카드 N장 · 배지 N개 · 티켓 N장)
- 메인: 타임라인 뷰 (최신순)
- 필터: 전체 / 티켓 / 기록 카드 / 배지

**진입 경로**
- 마이 메뉴 → 내 지갑 (노출 우선순위 상향)
- 기록 카드 생성 완료 시 "지갑에 추가됐어요" 토스트 → 탭 시 지갑으로 이동
- 홈 화면 내 지갑 진입 숏컷 추가

## 6. 비기능 요건

### 6-1. 데이터 안정성
- 유저 기록 데이터 클라우드 백업 필수
- 사진 서버 별도 구성, 장애 시 기록 텍스트 데이터는 유지
- 데이터 내보내기 기능 (v1.5 이후 고려)

### 6-2. 사진 정책
- MVP에서 기록당 사진 업로드 장수 제한 (구체적 수치 별도 정의)
- 추가 사진 업로드는 유료 옵션(포토패스) 테스트 고려
- 스토리지 비용 증가 선형 대응 방안 사전 설계

### 6-3. 성능
- GPS 인증 응답 시간 목표치 정의 필요
- 카드 생성 완료까지의 응답 시간 목표치 정의 필요

### 6-4. 보안
- 카드 고유 URL은 퍼블릭이나 유저 개인정보 노출 최소화
- 집관 기록의 보상 연동 시 허위 기록 방지 로직 별도 설계 필요 (v2 시점)

# PART 2. 고도화

## v1.5 — 출시 후 1개월 내

MVP 출시 후 초기 유저 리텐션 보완을 위해 빠르게 붙이는 기능들이다. 개발 착수는 MVP와 병렬로 준비하여 출시 후 지체 없이 배포한다.

### 과거 기록 소급 입력

**목적:** 기존 야구 팬의 "0에서 시작" 부담 제거, 초기 이탈 방지

**요건**
- 날짜 선택 + 팀 선택 시 해당 경기 결과 자동 매핑
- 직관 / 집관 여부 선택
- 소감, 사진 등 선택 입력 동일하게 제공
- 소급 입력 카드는 카드 내 "소급 입력" 표기 (신뢰도 구분)
- 입력 가능 기간 범위 정의 필요 (전 시즌까지 허용 여부)

### 구장 스탬프 지도

**목적:** 라이트 유저 포함 전 유저의 직관 동기부여, 수집 재미 제공

**요건**
- 전국 KBO 구장 지도 표시
- 직관 기록이 있는 구장에 스탬프 표기
- 전체 구장 방문 프로그레스 표시
- 미방문 구장 "아직 가보지 않은 구장이에요" 표기

## v2 — 출시 후 3개월 내외

유저 반응과 데이터를 기반으로 기능을 고도화하는 단계다.

### 감성 모드 / 기록 모드 이중 구조

**목적:** 야구 입문자와 헤비 유저를 동시에 만족시키는 기록 경험 제공

**요건**
- 기록 시작 시 모드 선택 (또는 설정에서 기본값 설정)
- 감성 모드: 날씨/기분/동행인/한 줄 소감 중심의 5개 이하 질문형 UI
- 기록 모드: 선발 투수, 인상적인 장면, 세부 통계 등 상세 입력 가능
- 모드 전환은 언제든 가능

### 카드 등급 세분화

**목적:** 이벤트 희소성에 따른 카드 가치 차별화, 수집 욕구 강화

**요건**
- 이벤트 생성 시 등급 설정 (일반 / 레어 / 에픽 / 레전더리)
- 등급별 카드 디자인 차별화
- 일반 직관 기록도 달성 조건에 따라 등급 부여 가능 (예: 연속 10회 직관)

### 배지 시스템 고도화

**목적:** 팬 활동 전반을 인정하는 보상 체계 구축

**요건**
- MVP의 응모 도전 배지 외 다양한 배지 추가
- 조건 예시: 첫 직관, 10회 직관, 연승 동행, 원정 직관, 전 구장 방문 등
- 배지도 지갑 내 컬렉션으로 표시

### 집관 실시간 퀴즈 인증

**목적:** 집관 기록 신뢰도 강화, 보상 연동 대비

**요건**
- 경기 중 특정 이벤트(이닝 종료, 득점 등) 트리거로 자동 팝업
- "현재 스코어는?" 형태의 간단한 확인 퀴즈
- 정답 시 집관 인증 강도 상향
- KBO 실시간 데이터 연동 전제

### 카드 홀로그램 인터랙션

**목적:** 카드의 물성 구현, NFT 느낌 강화

**요건**
- 카드 터치 시 홀로그램/광택 효과 애니메이션
- 기울기 센서 활용 빛 반사 효과 (가능 여부 기술 검토 필요)
- 카드 등급별 인터랙션 강도 차별화

### 응모 당첨 카드 타임라인 스토리

**목적:** 당첨 경험의 서사 강화

**요건**
- 응모일 → 당첨 발표일 → 직관 당일 타임라인 시각화
- 당첨 발표 푸시 알림 수신 타임스탬프 기록
- 카드 상세에서 스토리 보기 가능

### 시즌 Wrapped

**목적:** 시즌 마무리 시점의 공유 바이럴, 예측 이력만 있는 유저도 포함

**요건**
- 시즌 종료 후 자동 생성
- 포함 항목: 총 직관/집관 횟수, 직관 승률, 집관 승률, 가장 많이 간 구장, 예측 성공률, 시즌 하이라이트 경기
- 직관 기록 없는 유저도 예측 이력으로 Wrapped 생성 가능
- 풀스크린 슬라이드 형태, 각 슬라이드 독립 캡처/공유 가능
- 팀 컬러 기반 개인화 디자인

## 1. 월렛 고도화 단계

내 지갑은 단계적으로 폴리볼의 핵심 플랫폼 기능으로 성장한다.

### 1단계 — 지갑 리뉴얼 (MVP와 동시)
- 직관/집관 기록 카드 통합
- 고유 발행 번호 부여
- 카드 공유 URL 생성
- 타임라인 뷰

### 2단계 — 지갑 고도화 (v2 시점)
- 배지 시스템 본격화
- 카드 등급 세분화
- 홀로그램 인터랙션
- 카드 퍼블릭 페이지 고도화
- 첫 번째 확장 종목 추가 (파일럿)

### 3단계 — 월렛 플랫폼화 (장기)
- 다종목 / 다분야 카드 통합 관리
- 카드 선물하기 기능
- 파트너십 기반 카드 발행 (공연, 전시 등)
- NFT 전환 옵션 제공 (유저 선택)
- 카드 기반 멤버십 / 토큰 게이팅

## 2. 버티컬 확장 시나리오

직관 기록 카드의 데이터 구조는 처음부터 종목/분야 확장을 고려하여 설계한다. 카드 테이블에 카테고리 필드를 두어 야구 전용 구조를 피한다.

### 확장 순서 (안)

| 단계 | 분야 | 비고 |
|---|---|---|
| 현재 | KBO 야구 | MVP |
| 1차 확장 | K리그 또는 KBL | v2 파일럿, 구조 검증 |
| 2차 확장 | V리그, 기타 스포츠 | 1차 검증 후 |
| 3차 확장 | 공연 (뮤지컬/연극), 전시 | 감정 기반 기록으로 접근 |

### 확장 시 공통 구조

| 필드 | 내용 |
|---|---|
| 카테고리 | 야구 / 축구 / 농구 / 공연 / 전시 등 |
| 날짜 | 공통 |
| 장소 | 구장 / 공연장 / 전시관 |
| 이벤트명 | 경기명 / 공연명 / 전시명 |
| 결과 | 승패 (스포츠) / 감상 (문화) |
| 유저 입력 | 공통 (사진, 소감, 동행인) |

### 카드 디자인 언어

- 어떤 종목/분야가 추가되어도 "폴리볼 카드"라는 브랜드 프레임은 공통 유지
- 카테고리별 디자인 아이덴티티는 다르되, 폴리볼 브랜드가 먼저 인식되는 구조
- 비시즌 없는 지갑: 야구 비시즌에도 타 종목/공연 카드가 쌓여 지갑이 살아있는 상태 유지

## 3. 카드 자산화 로드맵

직관 기록 카드가 단순 아카이빙을 넘어 유저의 디지털 자산으로 인식되도록 단계적으로 구축한다.

### 현재 — 기록의 디지털화
- 경험을 카드 형태로 저장
- 고유 번호 부여, 공유 URL 생성

### 중기 — 자산 가치 부여
- 카드 등급 시스템으로 희소성 차별화
- 홀로그램 인터랙션으로 소유 감각 강화
- 배지 및 달성 시스템으로 수집 가치 부여

### 장기 — 자산의 유통
- 카드 선물하기 (소유권 이전)
- 파트너십 기반 카드 협업 발행
- NFT 전환 옵션 (유저 선택, 블록체인 기반 소유권 이전)
- 토큰 게이팅: 특정 카드/배지 보유 유저 대상 전용 혜택 제공
