# 폴리볼 개인정보 약관 자동화 봇 — 제품 스펙 문서 v1.1

> 작성일: 2026-05-11
> 목적: Claude Code 개발 핸드오프용 요구사항 정의서
> v1.1 변경: Phase 구조 재편(자동화 깊이 기준), 동의 거부 문구 변수화, 작성 지침 참조 추가

---

## 1. 프로젝트 개요

### 1-1. 배경 및 문제 정의

폴리볼 팀은 신규 기능을 기획할 때마다 개인정보 관련 약관(수집·이용 동의서, 개인정보처리방침 개정 등)을 작성하여 보안팀에 제출해야 한다. 현재 다음 세 가지 문제가 반복적으로 발생하고 있다.

**문제 1 — 작성 누락**: 기획 단계에서 개인정보 문서 작성 시점을 놓침. 디자인/개발이 착수된 이후 보안팀 제출 직전에 부랴부랴 작성하는 패턴이 반복됨.

**문제 2 — 판단 기준 부재**: 필수/선택 동의 구분, 수집 항목 범위, 보유기간 등을 작성자 주관으로 판단. 사람마다 결과물이 다름.

**문제 3 — 버전 파편화**: 작성자·시점별로 다른 양식/내용을 사용. 최신 개처방과 동기화되지 않는 문서가 존재.

### 1-2. 봇의 목적

기능정의서를 입력하면, 개인정보 관련 약관 문서를 법령 기준에 맞게 자동 생성하고, 기능정의서 내 동의 플로우 누락 여부를 체크하는 웹 기반 업무 자동화 도구.

### 1-3. 사용자

- 기획자: 기능정의서 입력 → 동의서 초안 획득 → 보안팀 제출
- 개발자: 생성된 동의서의 수집 항목 리뷰 → 누락/과잉 피드백
- 공통: 보안팀 검토 전 1차 셀프체크 용도

## 2. Phase 구조

Phase는 "기능 단위"가 아니라 **"자동화 깊이"**로 나눈다.

| Phase | 자동화 깊이 | 범위 | 핵심 산출물 |
|-------|-------------|------|------------|
| **Phase 1 (MVP)** | **생성 + 안내** | 동의서 자동 생성 + 개처방 개정 포인트 텍스트 안내 + 동의 플로우 체크 | 수집·이용 동의서 (docx) + 개처방 변경 안내 (텍스트) + 플로우 경고 |
| **Phase 2** | **생성 + 자동 반영** | 동의서 생성 시 개처방 개정안까지 자동 생성 + 버전 관리 + 제3자 제공 동의서 | 개처방 개정안 (docx) + 제3자 동의서 + 문서 동기화 |
| **Phase 3** | **관리 + 추적** | 히스토리 추적 + 보안팀 검토 인사이트 + 문서 간 정합성 자동 검증 | 변경 이력 대시보드 + 검토 인사이트 리포트 |

### Phase 전환 기준

Phase 1 → Phase 2 전환 조건:
- Phase 1을 팀이 **실제 기능정의서 2~3건에 적용**하여 실사용 검증 완료
- "동의서는 자동으로 나오는데 개처방 반영은 수동"이라는 불편이 확인됨
- Phase 1의 판정 정확도(필수/선택, 수집 항목)가 보안팀 검토를 통과하는 수준

Phase 2 → Phase 3 전환 조건:
- 누적 생성 문서가 일정량 이상 쌓여 히스토리 추적 필요성 발생
- 보안팀에서 검토 인사이트 자동화 요청

> 이하 Phase 1 상세 스펙. Phase 2~3은 Phase 1 검증 완료 후 별도 정의.

## 3. Phase 1 상세 스펙

### 3-1. 입력 (Input)

| 항목 | 설명 |
|------|------|
| 입력 형태 | 텍스트 붙여넣기 (기능정의서 본문) |
| 입력 포맷 | 플레인 텍스트 또는 HTML (기능정의서가 HTML로 작성되므로) |
| 입력 예시 | 기능정의서 전문 (예: 친구초대 기능정의서, 팀 응원전 기능정의서 등) |

### 3-2. 처리 로직

#### STEP 1 — 개인정보 수집 항목 분석

기능정의서를 분석하여 다음을 도출한다.

- **기능명**: 기능정의서의 제목/개요에서 추출
- **수집 목적**: 해당 기능이 개인정보를 왜 수집하는지 (1~2문장)
- **수집 항목**: 실제로 수집/이용/저장되는 개인정보 항목 목록
- **보유 기간**: 관련 법령 기반 보유기간 산정

**수집 항목 도출 기준**:
- 기능정의서 내 DB/API 설계 섹션의 컬럼/필드명 참조
- 화면 정의 내 입력 필드, 폼 요소 참조
- 유저 행동 기록 (참여 이력, 클릭 기록, 응모 이력 등) 포함
- 기기 정보, 접속 로그 등 자동 수집 항목은 기능 특성에 따라 판단

#### STEP 2 — 필수/선택 동의 판정

**판정 기준 (개보법 §15, §16 근거)**:

| 분류 | 기준 | 해당 기능 예시 |
|------|------|----------------|
| **필수** | 서비스 본질 기능 운영에 불가결. 이 항목이 없으면 서비스 자체를 제공할 수 없음 | 회원가입(계정 식별), 예측 참여(콘텐츠 이용기록), 응모/당첨(본인인증), 부정이용 방지(CI/DI) |
| **선택** | 부가 기능·이벤트·마케팅 목적. 동의하지 않아도 서비스 이용에 지장 없음 | 보상형 광고, 친구초대, 각종 이벤트, 팀 응원전, 커뮤니티, 마케팅 수신 |

**폴리볼 서비스 본질 기능 정의**:
- 회원가입/로그인 (계정 관리)
- 예측(픽) 참여 (핵심 콘텐츠)
- 응모권 획득/사용 (보상 체계)
- 티켓 응모/당첨 (최종 보상)
- 부정이용 방지 (서비스 안정성)

위 5가지에 직접적으로 필요한 수집 항목만 "필수". 나머지는 모두 "선택".

#### STEP 3 — 보유기간 산정

**기본 규칙**:
- 원칙: "회원 탈퇴 또는 동의 철회 시까지"
- 단, 관련 법령에 따른 보존 의무가 있는 경우 해당 기간 명시

**법령별 보존기간 참조 테이블**:

| 근거법령 | 보존 정보 | 기간 |
|----------|-----------|------|
| 전자상거래법 제6조 | 계약/청약철회 기록 | 5년 |
| 전자상거래법 제6조 | 대금결제/재화 공급 기록 | 5년 |
| 전자상거래법 제6조 | 소비자 불만/분쟁처리 기록 | 3년 |
| 전자상거래법 제6조 | 표시/광고 기록 | 6개월 |
| 통신비밀보호법 제15조의2 | 접속 로그 | 3개월 |
| 상법 제33조, 국세기본법 | 거래 장부 및 증빙 | 5년 |

**특수 케이스**: 부정이용 방지 목적의 경우 "탈퇴 시 별도 분리하여 90일간 보관 후 파기" 패턴 적용 (기존 동의서 참조)

#### STEP 4 — 기능정의서 내 동의 플로우 체크

기능정의서를 스캔하여 다음을 확인한다.

- 개인정보 동의 화면이 플로우에 포함되어 있는가?
- 동의 획득 시점이 정의되어 있는가? (최초 기능 사용 시, 회원가입 시 등)
- 동의 철회 경로가 명시되어 있는가?

**판정 결과**:
- ✅ 동의 플로우 존재 → 적정성 코멘트
- ⚠️ 동의 플로우 누락 → 경고 + 권장 동의 시점/방법 제안
- ℹ️ 별도 동의 불필요 → 기존 온보딩 약관으로 커버 가능한 경우 명시

### 3-3. 출력 (Output)

#### 출력 1 — 개인정보 수집·이용 동의서 (docx)

**파일명**: `폴리볼_개인정보_수집_및_동의서_{기능명}_{YYMMDD}.docx`

**문서 구조** (기존 템플릿 준수):

```
[헤더]
폴리볼 개인정보 수집 및 이용({기능명})

[본문 - 고정 문구]
(주)헥토이노베이션(이하 "회사") 「개인정보 보호법」 제15조 및 관계 법령이 정한 바에 따라,
폴리볼(PolyBall) 서비스(이하 "서비스") 제공을 위해 필요 최소한의 범위에서
개인정보를 수집·이용합니다.

처리하고 있는 개인정보는 아래의 목적 외의 용도로는 이용되지 않으며,
이용 목적이 변경되는 경우에는 「개인정보 보호법」 제18조에 따라
별도의 동의를 받는 등 필요한 조치를 이행할 예정입니다.

[소제목 - 고정 문구]
정보주체의 동의를 받아 처리하는 개인정보

회사는 「개인정보 보호법」 제15조제1항제1호에 따라 다음의 개인정보 항목을
정보주체의 사전 동의를 받아 처리하고 있습니다.

[테이블 - 봇이 채우는 영역]
| 구분 | 필수/선택 | 수집·이용 목적 | 수집·이용 항목 | 보유 및 이용 기간 |
|------|-----------|----------------|----------------|-------------------|
| {기능명} | {판정결과} | {목적} | {항목} | {기간} |

[동의 거부권 안내 - 반고정 문구]
위 개인정보 수집 및 이용에 동의하지 않으실 수 있으며,
동의하지 않으셔도 앱 이용은 가능합니다.
다만, 동의하지 않으실 경우 {불이익_내용}이 어려울 수 있습니다.

[버튼]
동의하기
```

**{불이익_내용} 생성 규칙**:
봇이 기능의 성격을 분석하여 동의 거부 시 불이익 내용을 자동 생성한다.
기존 동의서 패턴 및 기능 목적을 참조하되, 과도한 불이익 표현은 지양한다.

| 기능 유형 | 불이익 내용 예시 |
|-----------|-----------------|
| 이벤트/보상형 | 이벤트 참여 및 보상 제공 |
| 친구초대 | 친구 초대 이벤트 참여 및 초대 보상 수령 |
| 커뮤니티 | 커뮤니티 서비스 이용 |
| 마케팅 | 맞춤형 혜택 정보 수신 |
| 팀 응원전 | 팀 응원전 참여 및 주간 보상 수령 |

> 위 표는 예시이며, 봇은 기능정의서의 목적을 분석하여 적절한 불이익 문구를 자동 생성한다.

**테이블 컬럼별 작성 기준**:

| 컬럼 | 작성 기준 |
|------|-----------|
| 구분 | 기능명 (기능정의서 제목에서 추출) |
| 필수/선택 | STEP 2 판정 결과 |
| 수집·이용 목적 | 볼드체로 핵심 목적 기재. 쉼표로 구분. (예: **친구 초대 이벤트 참여 확인 및 보상 지급, 부정 참여 방지**) |
| 수집·이용 항목 | 구체적 데이터 항목 나열. (예: 이메일, 초대코드, 초대 관계 정보(초대자·피초대자 식별값), 보상 지급 이력) |
| 보유 및 이용 기간 | 볼드체로 기본 기간 + 단서 조항. (예: **회원 탈퇴 또는 동의 철회 시까지**) |

#### 출력 2 — 개처방 개정 필요 포인트 (Phase 2 범위이나 Phase 1에서 텍스트로 안내)

현행 개처방 제1조 제2항 "정보주체의 동의를 받고 처리하는 개인정보" 테이블과 대조하여:
- 신규 추가 필요 행이 있는지
- 기존 행의 수정이 필요한지
- 텍스트로 안내 (Phase 2에서 자동 개정안 생성)

#### 출력 3 — 동의 플로우 체크 결과

- 기능정의서 내 동의 플로우 존재 여부
- 누락 시: 권장 동의 시점 + 동의 화면 구성 제안
- 개발팀 확인 요청 사항 (추가 수집 항목 여부)

### 3-4. 개발팀 확인 프로세스

봇이 동의서를 생성한 후, 다음 안내를 함께 출력한다.

```
📋 개발팀 확인 요청

아래 수집 항목이 실제 구현과 일치하는지 확인해주세요.

[봇이 도출한 수집 항목 리스트]

확인 사항:
1. 위 항목이 맞나요?
2. 추가로 수집하는 데이터가 있나요?
3. 위 항목 중 실제로는 수집하지 않는 것이 있나요?

→ 피드백을 입력하면 동의서를 업데이트합니다.
```

개발자가 피드백을 입력하면 봇이 동의서를 재생성한다.

## 4. 참조 문서 목록

봇이 판단 근거로 사용하는 참조 문서. Claude Code 구현 시 이 문서들을 봇의 컨텍스트에 포함시켜야 한다.

| 문서명 | 용도 | 비고 |
|--------|------|------|
| 폴리볼 법령 레퍼런스 v1.0 | 법령 기반 판정 기준 | 개보법 §15~§30, 정통망법 §22의2, §50 |
| 동의서 템플릿 (보상형광고/이벤트/친구초대) | 출력 양식 기준 | docx 3종 |
| 개인정보처리방침 V3 | 현행 개처방 대조용 | 제1조 제2항 테이블 기준 |
| 폴리볼 서비스 IA | 서비스 구조 이해 | 기능 간 관계 파악용 |
| 개인정보 처리방침 작성 지침 (개인정보보호위원회) | 문서 작성 표준 | Phase 1: 용어/표현 정합성, Phase 2: 개처방 개정안 구조 기준 |

> **개인정보 처리방침 작성 지침 활용 범위**:
> - Phase 1에서는 동의서의 용어·표현이 작성 지침의 표준과 일치하는지 크로스체크 용도로 활용
> - Phase 2에서는 개처방 개정안 생성 시 필수 기재 항목, 라벨링 방식, 문서 구조의 직접적 기준으로 활용
> - 팀에서 작성 지침 파일을 보유하고 있으면 봇 참조 문서에 포함, 없으면 핵심 원칙을 시스템 프롬프트에 요약 내장

## 5. 필수/선택 판정 가이드라인

봇의 핵심 판단 로직. 이 가이드라인을 시스템 프롬프트 또는 판정 규칙으로 내장한다.

### 5-1. 판정 플로우차트

```
기능정의서 입력
    │
    ▼
해당 기능이 서비스 본질 기능인가?
    │
    ├─ YES → 해당 수집 항목이 본질 기능 운영에 불가결한가?
    │           ├─ YES → "필수"
    │           └─ NO  → "선택"
    │
    └─ NO  → "선택"
```

### 5-2. 폴리볼 서비스 본질 기능 목록 (필수 동의 해당 가능)

1. 회원가입/로그인 — 계정 식별, 가입자 관리
2. 예측(픽) 참여 — 콘텐츠 이용, 정답 판정
3. 응모권 획득/사용 — 보상 체계 운영
4. 티켓 응모/당첨 — 최종 보상 제공, 본인인증
5. 부정이용 방지 — 서비스 안정성 확보

### 5-3. 선택 동의 해당 기능 (부가 기능)

이벤트, 친구초대, 팀 응원전, 보상형 광고, 커뮤니티, 마케팅, 프로모코드, 알림 수신 등. 이 목록은 신규 기능 추가 시 확장됨.

## 8. Claude Code 작업 요청 가이드

### 8-1. Claude Code란

터미널(명령줄)에서 Claude에게 코딩 작업을 시키는 Anthropic의 개발 도구. 이 스펙 문서를 기반으로 실제 웹 앱 코드를 생성할 때 사용.

### 8-2. Claude Code에 요청하는 방법

**원칙: 한 번에 다 시키지 말고, 단계별로 나눠서 요청한다.**

아래 순서로 진행하면 효율적이다.

#### 요청 1 — 프로젝트 셋업

```
이 스펙 문서를 읽고 프로젝트 구조를 잡아줘.
- 기술 스택: [확정된 스택]
- 프로젝트명: polyball-privacy-bot
- 폴더 구조와 주요 파일 목록만 먼저 만들어줘
```

#### 요청 2 — 핵심 로직 구현

```
STEP 1~4 분석 로직을 구현해줘.
- 입력: 기능정의서 텍스트
- Anthropic API를 호출해서 분석 수행
- 시스템 프롬프트에 필수/선택 판정 가이드라인과 보유기간 규칙을 포함
- 출력: JSON 형태의 분석 결과
참조: 스펙 문서 섹션 3-2, 5, 6
```

#### 요청 3 — 동의서 docx 생성

```
분석 결과 JSON을 받아서 워드 동의서를 생성하는 기능을 만들어줘.
- docx 라이브러리 사용
- 템플릿 구조는 스펙 문서 섹션 3-3의 "출력 1" 참조
- 기존 동의서 샘플 3종의 양식을 정확히 따를 것
```

#### 요청 4 — 프론트엔드 UI

```
웹 UI를 만들어줘.
- 메인: 텍스트 입력 + 분석 버튼
- 결과: 분석 요약 + 동의서 미리보기 + 다운로드 + 플로우 체크 결과
- 피드백: 개발자 피드백 입력 → 재생성
- 스펙 문서 섹션 7 참조
```

#### 요청 5 — 테스트

```
기존 기능정의서(친구초대, 팀 응원전, 보상형 광고)로 테스트해줘.
- 각각 동의서를 생성하고
- 기존에 작성된 실제 동의서와 비교해서
- 수집 항목, 필수/선택 판정, 보유기간이 일치하는지 확인
```

### 8-3. Claude Code 사용 팁

1. **CLAUDE.md 파일을 만들어라**: 프로젝트 루트에 CLAUDE.md를 만들어서 프로젝트 컨텍스트, 규칙, 참조 문서 위치를 적어두면 Claude Code가 매 대화마다 이를 자동으로 읽는다. 이 스펙 문서의 핵심 내용을 CLAUDE.md에 요약해두면 좋다.

2. **참조 문서를 프로젝트 폴더에 넣어라**: 법령 레퍼런스, 동의서 템플릿, 개처방 파일을 프로젝트 내 `/docs` 폴더에 복사해두면 Claude Code가 직접 읽을 수 있다.

3. **한 번에 하나씩 요청하라**: "전부 다 만들어줘"보다 "프로젝트 구조 잡아줘" → "분석 로직 만들어줘" → "UI 만들어줘" 순서가 결과물 품질이 훨씬 높다.

4. **에러가 나면 에러 메시지를 그대로 붙여넣어라**: Claude Code는 에러 메시지를 보면 바로 수정할 수 있다. 설명하려 하지 말고 그대로 복사해서 주면 된다.

5. **중간중간 확인하라**: 한 단계 끝날 때마다 실행해보고 결과를 확인한 후 다음 요청으로 넘어가라.

6. **되돌리기를 두려워하지 마라**: Claude Code는 git과 함께 쓰면 좋다. 각 단계마다 커밋해두면 잘못된 방향으로 갔을 때 쉽게 되돌릴 수 있다.

## 9. Phase 2~3 예정 범위 (참고)

### Phase 2 — 생성 + 자동 반영
- 동의서 생성 시 개처방 제1조 제2항 테이블 자동 대조 → 개정안(docx) 자동 생성
- 개인정보 처리방침 작성 지침 기반으로 개처방 구조/필수 기재 항목 검증

- 기능별 약관 개정 히스토리 추적 (누가, 언제, 어떤 기능에서 어떤 항목이 변경되었는지)

- 문서 간 정합성 자동 검증 (동의서 ↔ 개처방 ↔ 기능정의서 3자 크로스체크)
