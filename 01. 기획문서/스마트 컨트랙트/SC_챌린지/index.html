왜 이 챌린지를 SC로 만들어야 하는가
시장 검증 → Web2 한계 → SC 전환 논리
BACKGROUND
① 시장 검증 — "돈을 거는 행위가 목표 달성률을 높인다"는 건 이미 증명됐다
서약 계약(Commitment Contract) 모델은 국내외에서 이미 동작하는 비즈니스입니다. 단순 의지에만 의존하지 않고 금전적 페널티·보상을 결합했을 때 목표 달성률이 유의미하게 높아진다는 것은 아래 레퍼런스들을 통해 검증됐습니다.
국내
🇰🇷
챌린저스 (Challengers)
목표를 설정하고 참가비를 예치. 달성 실패 시 벌금이 성공자에게 분배되는 구조로 국내 서약 계약 모델을 검증한 서비스. 높은 목표 달성률로 모델 유효성을 증명했으나, 이후 뷰티 체험단 BM으로 피벗.
해외
StickK 미국
예일대 교수진 설립. 체중 감량·금연 등 목표에 돈을 거는 서약 계약 시스템. 심판(Referee) 지정 시 목표 달성률 약 80% 달성.
GymPact 미국
운동 목표 달성 특화. 현금 보상으로 실제 운동을 독려. 약속한 운동 목표의 90% 달성률 기록.
HealthyWage 미국
다이어트 특화. "살을 빼서 돈을 획득하라"는 슬로건으로 체중 감량 목표에 금전적 인센티브를 직접 결합.
Weight Wins 영국
서약 접근법이 일반적인 다이어트 방법보다 성공적이라는 연구 결과 기반. 목표 달성을 금전 서약으로 지원.
21Habit 글로벌
"21일 안에 습관을 만들거나 끊기" 모토. 단기 목표와 습관 형성에 특화된 서약 계약 플랫폼.
이 외 Strava, Mindbloom, FitID, Health Month 등 GPS 트래킹·소셜 지원 결합 서비스 다수 운영 중.
② Web2의 구조적 한계 — 그런데 왜 성장이 막혔나
모델은 검증됐지만 Web2 구조에서는 세 가지 근본적인 한계가 존재합니다. 챌린저스가 뷰티 체험단으로 피벗한 것도 이 한계의 결과입니다.
⚔️
이해상충 구조
유저가 실패해야 플랫폼이 돈을 버는 구조. 유저 성공률이 높아질수록 플랫폼 수익이 악화. 서비스와 유저의 목표가 근본적으로 충돌.
🏦
플랫폼 신뢰 의존
유저의 예치금이 회사 계좌에 보관. 플랫폼 파산·횡령 시 예치금 회수 불가. 유저는 규칙이 아닌 회사를 믿어야 하는 구조.
💸
운영 비용 폭발
사진 인증 검수 인건비, 정산 CS, 분쟁 처리 비용이 규모가 커질수록 폭발적으로 증가. 사람이 개입해야 돌아가는 구조의 한계.
③ SC로의 전환 — 세 가지 한계를 구조적으로 해결한다
스마트 컨트랙트는 단순히 Web2를 Web3로 옮기는 것이 아닙니다. Web2의 구조적 한계를 코드 레벨에서 제거합니다.
이해상충 해결
유저가 성공해도 플랫폼이 수익을 낸다
예치금이 DeFi 프로토콜에 자동 예치되어 이자를 생산. 유저 성공 여부와 무관하게 예치 기간 동안 이자가 발생하고 플랫폼은 이 이자의 일부를 수수료로 수취.
신뢰 의존 해결
회사를 믿는 것에서 코드를 믿는 것으로
유저의 예치금은 회사 계좌가 아닌 SC에 보관. 조건이 충족될 때만 코드가 자동 실행. 플랫폼이 임의로 개입하거나 자금을 유용하는 것이 구조적으로 불가능.
운영 비용 해결
사람 없이 돌아가는 정산 구조
오라클 기반 자동 검증으로 인증 검수 인건비 제거. SC 자동 정산으로 분쟁·CS 비용 구조적 감소. 규모가 커져도 운영 비용이 선형적으로 늘지 않음.
템플릿 3종
🎯
Type 1 · 1인 목표 달성
개인 챌린지
호스트가 목표를 설정하고 스테이킹. 게스트가 응원 스테이킹으로 참여하는 구조
✅ 초기 론칭 추천
IF 호스트가 목표·기간을 설정하고 스테이킹. 게스트가 응원 스테이킹으로 참여
→
THEN 달성 시 호스트 원금+이자 회수. 응원 게스트에게도 이자 분배
플랫폼 이익
  • DeFi 이자 수수료 일부 귀속
  • 템플릿 개설 수수료
  • TVL 증가 → 이자 풀 자동 확대
유저 가치
  • 목표 달성 동기 강화 (스테이킹 압박)
  • 응원 구조 → 소셜 요소 내재화
  • 성공 시 이자 보너스 수령
  • 게스트도 응원으로 이자 수익
SC여야 하는 이유
응원 게스트 이자 분배를 중간자 없이 자동 실행.

Web2에선 호스트가 응원금을 직접 관리해야 해서 신뢰 문제 발생. SC가 이를 구조적으로 차단.
검증 방식
앱 내 행동 외부 API
걸음수, 운동 기록, 수면 등
헬스앱 API 연동 우선

오라클 의존도 낮음
✅ Safe — 원금 보장
원금 전액 반환 + 이자만 상금. 달성 실패 시에도 원금 돌려받음. 참여 허들 낮음. 법적 리스크 낮음.
⚠️ Risk — 원금 손실 가능
미달성 시 스테이킹 일부 손실. 참여 동기 강하나 법무 검토 필요.
참여 구조
호스트 1인 + 응원 게스트 N명 (인원 무제한)
이자 재원
스테이킹 기간 DeFi 수익 (Aave, Compound 등)
개발 복잡도
낮음 구조 단순, 초기 구현 용이
👥
Type 2 · 다수 공동 참여
그룹 챌린지
N명이 동일 조건으로 참여. 성공자가 실패자 이자를 가져가는 구조
✅ 초기 론칭 추천
IF N명이 동일 조건으로 스테이킹 후 챌린지 참여 (최소/최대 인원 설정)
→
THEN 성공자 원금+이자 반환. 실패자 이자가 성공자 풀로 자동 재분배
플랫폼 이익
  • DeFi 이자 수수료 일부 귀속
  • 참여자 수 비례 TVL 증가
  • 그룹 구조 → 바이럴 효과로 신규 유저 유입
유저 가치
  • 집단 압박 효과 → 목표 완수율 향상
  • 성공 시 실패자 이자까지 추가 수익
  • 혼자선 흐지부지되는 목표를 그룹으로 완수
  • 참여자 多 → 이자 풀 자동 증가
SC여야 하는 이유
성공/실패 판정과 이자 재분배를 완전 자동화.

Web2에선 운영자가 수동 정산해야 하고 조작 가능성 존재. SC가 판정과 분배를 코드로 확정.
검증 방식
앱 내 행동
개인별 성공/실패 자동 판정
걸음수, 운동 기록 등

오라클 의존도 낮음
✅ Safe — 원금 보장
원금 전액 보장. 실패자 이자만 성공자에게 이동. 모두 성공하면 전원 원금+이자 수령.
⚠️ Risk — 원금 손실 가능
실패자 원금 일부가 성공자 풀에 포함. 상금 규모 커지나 법무 검토 필요.
참여 구조
동일 조건 N명 (최소/최대 인원 호스트 설정)
이자 재원
스테이킹 기간 DeFi 수익 + 실패자 이자 재분배
개발 복잡도
낮음 개인 챌린지 구조 확장으로 구현
🏆
Type 3 · 경쟁 순위 기반
대회 / 토너먼트
참가비를 SC가 보관하고 순위에 따라 상금 풀+이자를 자동 분배하는 구조
🔶 초기엔 조건부
IF 참가자 N명이 참가비를 스테이킹하고 대회에 참여 (동호회, 해커톤, 스포츠 기록 등)
→
THEN 순위 확정 후 상위 N명에게 전체 풀+이자 차등 분배. 탈락자 원금 반환(보장 버전)
플랫폼 이익
  • 참가비 풀 이자 수수료
  • 대회 개설 수수료
  • 참가자 多 → TVL·이자 자동 증가
유저 가치
  • 공정한 실력 경쟁 보장
  • 참가자 많을수록 상금 풀 자동 증가
  • 주최자 먹튀·상금 미지급 분쟁 구조적 차단
  • 소규모 동호회도 안전한 대회 운영 가능
SC여야 하는 이유
참가비를 주최자가 아닌 SC가 보관.

Web2에선 주최자 계좌로 입금 → 먹튀·미지급 분쟁 실제 발생. SC는 조건 충족 시에만 자동 지급.
검증 방식
앱 내 기록 외부 API 상호 인증
종목에 따라 선택 적용

오라클 의존도 중간
✅ Safe — 원금 보장
참가비 전액 환불 + 이자만 상금 풀. 참여 허들 낮고 대회 흥행 목적에 유리. 법적 리스크 낮음.
⚠️ Risk — 원금 손실 가능
참가비 전액이 상금 풀. 우승 시 대형 보상. 실력 기반 설계 시 리스크 완화 가능. 법무 검토 필수.

이자 재원
참가비 풀 전체 DeFi 예치 → 이자 발생
개발 복잡도
