Self Commit Contract · 1:1 구조
의지력, 신뢰, 분쟁 — 기존에는 사람에게 의존했던 것들을 스마트 컨트랙트가 구조적으로 해결합니다. 조건이 충족되면 자동 실행, 충족되지 않으면 실행 불가. 중개자도, 예외도 없습니다.

셀프 커밋 컨트랙트의 핵심은 "당사자 간 신뢰를 코드로 대체"하는 것입니다. 나 자신과의 약속(저축·지출), 상대방과의 약속(정산·보증금) 모두 조건이 충족되기 전까지 누구도 자금에 접근할 수 없는 구조로 설계됩니다. Web2에서는 "해지 요청" 한 번으로 모든 약속이 깨질 수 있었습니다.

Ideas — 3 Concepts
01
프리랜서 마일스톤 정산
"다 해줬는데 돈 안 준다" — 플랫폼 없이 계약을 강제하다
에스크로프리랜서자동 정산글로벌 검증
배경 & 문제
국내 프리랜서 시장(디자이너·개발자·작가)에서 "작업 완료 후 대금 미지급" 분쟁은 구조적으로 반복됩니다. 크몽·숨고 같은 플랫폼이 중재하지만, 플랫폼 없는 직거래에서는 신뢰 외에 다른 보장 수단이 없습니다. 의뢰인 입장에서도 선불 후 먹튀 위험이 동일하게 존재합니다.
해외 유사 서비스
Zenland프리랜서 대금을 스마트 컨트랙트에 예치 → 결과물 승인 시 즉시 지급. 수수료 1% (최대 $50 상한). 비수탁형 구조로 플랫폼이 자금 보관 안 함.
Kleros Escrow분쟁 발생 시 익명의 크라우드소싱 배심원단이 판결. 중앙화 고객센터 없이 탈중앙 분쟁 해결 구현.
작동 방식
01 →
의뢰인이 전체 금액을 컨트랙트에 잠금
02 →
마일스톤 단계별 납품 & 확인 (1차 시안 → 30%, 최종 납품 → 70%)
03 →
각 단계 확인 즉시 해당 금액 자동 송금
04 →
분쟁 발생 시 → 중재 모듈 또는 배심원 시스템 작동
Web2 대비 차별점
기존 Web2
✕플랫폼 수수료 10~20%
✕플랫폼 정책에 따라 결과 달라짐
✕직거래 시 보장 수단 없음
스마트 컨트랙트
✓수수료 1% 이하
✓조건 충족 시 자동 실행, 개입 불가
✓플랫폼 없는 직거래에서도 신뢰 보장
성공 체크 기준 (오라클)
납품 확인의뢰인의 온체인 서명(승인) → 컨트랙트가 인식하여 자동 지급 트리거
기간 오라클일정 기간 내 이의 없으면 자동 승인 처리 (예: 7일 이내 미이의 시 자동 지급)
분쟁 오라클이의 제기 시 → 배심원/중재 모듈이 판결 결과를 컨트랙트에 전달
리스크 / 고려사항
⚠결과물 품질 판단은 코드로 자동화 불가 → 분쟁 해결 모듈 설계가 핵심 난제
⚠국내 전자금융거래법 및 에스크로 규제 검토 필요
02
개인 적금 / 지출 컨트랙트
"나 자신과의 계약" — 의지력을 코드로 강제하다
강제 저축지출 관리Commitment Device월렛 온보딩
배경 & 문제
사회초년생·20대를 중심으로 "저축하고 싶지만 지켜지지 않는" 문제는 보편적입니다. 카카오뱅크 세이프박스, 은행 아이적금 같은 상품이 있지만, 결국 "해지 요청하면 풀린다"는 한계가 있습니다. 행동경제학에서는 이를 Commitment Device(사전 구속 장치)로 정의합니다. 미래의 내가 유혹에 흔들리지 않도록, 현재의 내가 구조적으로 막아두는 방식입니다.
해외 유사 개념 & 서비스
StickK목표 미달 시 원하지 않는 단체에 자동 기부되는 Web2 Commitment 서비스. 행동경제학 기반. 단, 자금 잠금은 불완전.
DeFi 적금스마트 컨트랙트로 일정 기간 자금을 완전 잠금. 조건 미충족 시 인출 자체가 코드 수준에서 불가능한 구조.
작동 방식
01 →
사용자가 조건과 금액을 직접 설정 (예: 6개월 잠금, 월 30만원)
02 →
컨트랙트에 자금 잠금 — 조건 충족 전까지 인출 불가
03 →
지출 목표 연동 시 → 초과 지출 감지되면 다음 달 자동 차감 또는 벌금 적용
04 →
조건 달성 시 → 적립금 자동 반환 (보너스 옵션 가능)
Web2 대비 차별점
기존 Web2
✕언제든 해지 가능 (의지력에 의존)
✕은행 영업시간·정책 의존
✕조건 커스터마이징 불가
스마트 컨트랙트
✓코드 수준에서 인출 불가 — 진짜 강제
✓조건을 내가 직접 설계
✓월렛 서비스 첫 진입점으로 활용 가능
성공 체크 기준 (오라클)
시간 오라클설정한 기간(예: 6개월) 경과 여부를 블록체인 타임스탬프로 자동 확인
지출 오라클카드사 API 연동 → 월 지출액 데이터를 컨트랙트에 전달, 초과 여부 자동 판단
비상 해제긴급 상황 시 멀티 서명(본인 + 보증인 동의) 조건으로 예외 인출 가능
리스크 / 고려사항
⚠긴급 상황 시 인출 불가 → 비상 해제 조건 설계 필수
⚠지출 데이터 오라클 연동 필요 (카드사 API 등) — 초기 구현 복잡도 높음
03
보증금 세이프 볼트
전월세 보증금을 금고에 잠그고, 집주인에겐 이자를 — 세입자에겐 안전을
전월세보증금 보호DeFi 이자HUG 대체
배경 & 문제
전세 보증금 미반환 사고는 매년 수천 건 발생합니다. 현행 주택도시보증공사(HUG) 전세보증보험은 "사고 난 후 보상" 구조로, 보증금을 되찾는 데 수개월이 걸립니다. 근본 문제는 보증금이 집주인 개인 자산으로 넘어가기 때문입니다. 집주인이 보증금을 다른 곳에 사용하거나 채무 문제가 생기면 세입자는 속수무책입니다.
해외 유사 사례
영국 TDS임대차 보증금을 정부 지정 제3기관이 보관하는 의무 예치 제도. 계약 만료 시 자동 반환. 스마트 컨트랙트로 완전 자동화 가능한 구조.
PoolTogether예치 자금을 DeFi 풀에 운용해 이자를 생성하는 구조. 원금 보장 + 이자 분배 메커니즘의 선행 사례.
작동 방식
01 →
세입자가 보증금을 컨트랙트(금고)에 예치 — 집주인 직접 수취 불가
02 →
집주인이 DeFi 운용 동의 시 → 원금을 DeFi 풀에 예치, 이자 발생
03 →
계약 기간 중 원금은 인출 불가 — 이자만 집주인에게 주기적 지급
04 →
계약 만료 시 → 원금 세입자 자동 반환 / 파손 분쟁 시 소액 보험료로 보상 처리
Web2 대비 차별점
기존 Web2 (HUG 보험)
✕사고 후 보상 — 수개월 소요
✕보험료 부담 (세입자)
✕집주인 자산 운용 통제 불가
스마트 컨트랙트
✓사고 자체가 불가 — 원금이 컨트랙트 밖으로 못 나감
✓집주인은 DeFi 이자 수익으로 참여 인센티브 확보
✓계약 만료 시 자동 반환 — 분쟁 없음
성공 체크 기준 (오라클)
계약 만료 오라클임대차 계약 종료일을 블록체인 타임스탬프로 확인 → 자동 반환 트리거
파손 분쟁 오라클양측 서명 불일치 시 → 소액 보증금(보험료)에서 중재 비용 자동 처리
DeFi 이자 오라클운용 풀의 이자 발생량을 주기적으로 컨트랙트에 전달 → 집주인 자동 지급
리스크 / 고려사항
⚠집주인의 자발적 참여가 전제 — 참여 인센티브(이자) 설계가 핵심
⚠임대차보호법상 보증금 예치 의무화 여부 법률 검토 필요
⚠DeFi 풀 운용 리스크 (원금 손실 가능성) → 안전 자산 한정 운용 필요
